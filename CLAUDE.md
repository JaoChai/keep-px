# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Keep-PX (codebase name: Pixlinks) is a Facebook Pixel data preservation and replay platform. It captures Facebook Pixel events through a JavaScript SDK, stores them in a database, forwards them to Facebook Conversions API (CAPI), and allows replaying stored events to a new pixel when accounts get banned.

## Monorepo Structure

Three independent packages in one repo ‚Äî no shared workspace tooling:

- **`backend/`** ‚Äî Go REST API (chi router, pgx/v5, Neon PostgreSQL)
- **`frontend/`** ‚Äî React SPA dashboard (Vite, TanStack Query, Zustand, Tailwind CSS v4)
- **`sdk/`** ‚Äî Browser JavaScript SDK for event tracking (Rollup, TypeScript)

## Build & Run Commands

### Backend (Go)
```bash
cd backend
cp .env.example .env          # configure DATABASE_URL, JWT_SECRET
go run ./cmd/server            # start dev server on :8080
go build -o server ./cmd/server # build binary
go test ./...                  # run all tests
go test ./internal/service/... # run specific package tests
```

### Frontend (React/Vite)
```bash
cd frontend
npm install
npm run dev                    # dev server on :5173 (proxies /api to :8080)
npm run build                  # tsc + vite build
npm run lint                   # eslint
```

### SDK (TypeScript/Rollup)
```bash
cd sdk
npm install
npm run build                  # outputs dist/pixlinks.min.js (UMD) + dist/pixlinks.esm.js
npm run dev                    # watch mode
```

### Database Migrations
SQL migrations are in `backend/db/migrations/`. Apply manually against Neon PostgreSQL. No migration CLI tool is configured ‚Äî run the `.up.sql` files directly.

### sqlc Code Generation
```bash
cd backend
sqlc generate                  # regenerates backend/db/generated/ from db/queries/*.sql
```
**Do not manually edit files in `backend/db/generated/`** ‚Äî they are auto-generated by sqlc.

## Development Workflow

### Branch Rules
- NEVER commit directly to `main`. Always create a feature branch.
- Branch naming: `feat/`, `fix/`, `chore/`, `refactor/`
- One logical change per branch. Keep PRs focused.

### Before Writing Code
- Non-trivial tasks (new features, multi-file changes): use plan mode first.
- New backend resources: use `/go-service-scaffold`.
- New endpoints on existing resources: use `/api-endpoint`.
- Database schema changes: use `/db-migration` first.
- New frontend pages: use `/frontend-feature`.

### Quality Gates (MUST Pass Before Push)
Backend:
```bash
cd backend && go vet ./... && go test -race ./...
```
Frontend:
```bash
cd frontend && npm run lint && npm run test && npm run build
```
SDK:
```bash
cd sdk && npm run test && npm run build
```
Run only gates for packages you changed. Do NOT push code that fails.

### Code Review (MUST Run Before Creating PR)
- Backend changes: run `/go-review`.
- Changes touching auth, middleware, user input, or API keys: run `/security-review`.

### Pull Requests
- Create via `gh pr create`. Use conventional commit title (e.g., `feat: add webhook support`).
- Squash merge only. Include Summary and Test Plan in PR body.
- Wait for `ci-gate` to pass before merging.

### After Merge
- Check `deploy-verify` job in GitHub Actions to confirm deploy succeeded.
- If failed, check Railway logs immediately.

### Commit Messages
Format: `type: short description` (lowercase, no period, max 72 chars)
Types: `feat`, `fix`, `chore`, `refactor`, `ci`, `docs`, `test`

## Architecture

### Backend ‚Äî Clean Architecture (Go)

Dependency flow: `handler ‚Üí service ‚Üí repository ‚Üí database`

- **`cmd/server/main.go`** ‚Äî Entry point. Wires up config, pgxpool, router, graceful shutdown.
- **`internal/config/`** ‚Äî Env-based config using `caarlos0/env`. Loads `.env` via godotenv.
- **`internal/domain/`** ‚Äî Pure domain structs (Customer, Pixel, PixelEvent, EventRule, ReplaySession). No dependencies.
- **`internal/repository/interfaces.go`** ‚Äî Repository interfaces (CustomerRepository, PixelRepository, EventRepository, etc.).
- **`internal/repository/postgres/`** ‚Äî pgx/v5 implementations. Hand-written SQL (not using sqlc-generated code in repos ‚Äî sqlc is set up but repos use direct queries).
- **`internal/service/`** ‚Äî Business logic. Each service takes repository interfaces. Sentinel errors (ErrPixelNotFound, etc.) for handler error mapping.
- **`internal/handler/`** ‚Äî HTTP handlers. Use `go-playground/validator` for input validation. Consistent response via `handler.JSON()` / `handler.ErrorJSON()` with `APIResponse` / `PaginatedResponse` structs.
- **`internal/middleware/`** ‚Äî JWT auth (Bearer token, HMAC-SHA256), API key auth (X-API-Key header), CORS, request ID, structured logging.
- **`internal/router/router.go`** ‚Äî chi router. All DI wiring happens here (repos ‚Üí services ‚Üí handlers).
- **`internal/facebook/capi.go`** ‚Äî Facebook Conversions API HTTP client.

### Authentication ‚Äî Dual Mode
- **Dashboard (JWT)**: `POST /api/v1/auth/register|login|refresh` ‚Üí Bearer token. Customer ID stored in JWT `sub` claim, extracted via `middleware.GetCustomerID(ctx)`.
- **SDK (API Key)**: `X-API-Key` header ‚Üí validated against `customers.api_key` column. Used only for event ingestion endpoint.

### Key Patterns
- Ownership verification: Services always check `pixel.CustomerID == customerID` before operations.
- CAPI forwarding: Events are forwarded to Facebook asynchronously via `go s.forwardToCAPI(...)`.
- Replay: Creates a background goroutine with semaphore-limited concurrency (5 workers), ~50 events/sec rate limit, progress tracking.
- Repository nil convention: `GetByID` returns `(nil, nil)` when not found (not an error), callers check for nil.

### Frontend Architecture
- **State**: Zustand for auth state, TanStack Query for server state
- **Routing**: react-router v7 with `ProtectedRoute` wrapper
- **API client**: Axios with auto token refresh interceptor (`src/lib/api.ts`)
- **Path alias**: `@/` maps to `frontend/src/`
- **UI components**: shadcn/ui pattern in `src/components/ui/`
- **Vite proxy**: `/api` requests proxy to `http://localhost:8080` in dev

### SDK Architecture
- `PixlinksTracker`: Batched event queue (flush every 2s or at 10 events), retry up to 3 times, sends to `/api/v1/events/ingest`
- `VisualSetup`: Overlay-based element selector for visual event configuration, communicates with dashboard via postMessage
- Auto-init from `<script data-pixlinks-key="...">` tag

| GET | `/api/v1/events` | JWT | Event log (paginated) |
| GET/POST/PUT/DELETE | `/api/v1/pixels/{pixelId}/rules/*` | JWT | Event rules |

| GET | `/api/v1/analytics/*` | JWT | Dashboard analytics |
| GET | `/api/v1/proxy?url=` | JWT | HTML proxy for visual setup |

## Database

PostgreSQL on Neon. 6 tables: `customers`, `pixels`, `pixel_events`, `event_rules`, `replay_sessions`, `refresh_tokens`. All IDs are UUID with `gen_random_uuid()`. Timestamps are `TIMESTAMPTZ`.

## Deployment

Railway with separate services:
- **Backend**: `backend/Dockerfile` ‚Äî Multi-stage Go build, Alpine-based
- **Frontend**: `frontend/Dockerfile` ‚Äî Node build ‚Üí Nginx serving static files. Requires `VITE_API_URL` build arg.

## Environment Variables

Backend (see `backend/.env.example`):
- `DATABASE_URL` (required) ‚Äî Neon PostgreSQL connection string
- `JWT_SECRET` (required) ‚Äî HMAC signing key
- `PORT`, `ENV`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`, `FB_GRAPH_API_URL`, `CORS_ALLOWED_ORIGINS`, `RATE_LIMIT_RPS`

Frontend:
- `VITE_API_URL` ‚Äî Backend API base URL (empty string uses relative path with Vite proxy in dev)
