# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Keep-PX (codebase name: Pixlinks) is a Facebook Pixel data preservation and replay platform. It captures Facebook Pixel events through built-in sale page templates, stores them in a database, forwards them to Facebook Conversions API (CAPI), and allows replaying stored events to a new pixel when accounts get banned.

## Monorepo Structure

Two independent packages in one repo — no shared workspace tooling:

- **`backend/`** — Go REST API (chi router, pgx/v5, Neon PostgreSQL)
- **`frontend/`** — React SPA dashboard (Vite, TanStack Query, Zustand, Tailwind CSS v4)

## Build & Run Commands

### Backend (Go)
```bash
cd backend
cp .env.example .env          # configure DATABASE_URL, JWT_SECRET
go run ./cmd/server            # start dev server on :8080
go build -o server ./cmd/server # build binary
go test ./...                  # run all tests
go test ./internal/service/... # run specific package tests
```

### Frontend (React/Vite)
```bash
cd frontend
npm install
npm run dev                    # dev server on :5173 (proxies /api to :8080)
npm run build                  # tsc + vite build
npm run lint                   # eslint
```

### Database Migrations
SQL migrations are in `backend/db/migrations/`. Apply manually against Neon PostgreSQL. No migration CLI tool is configured — run the `.up.sql` files directly.

### sqlc Code Generation
```bash
cd backend
sqlc generate                  # regenerates backend/db/generated/ from db/queries/*.sql
```
**Do not manually edit files in `backend/db/generated/`** — they are auto-generated by sqlc.

## Development Workflow

### Branch Rules
- NEVER commit directly to `main`. Always create a feature branch.
- Branch naming: `feat/`, `fix/`, `chore/`, `refactor/`
- One logical change per branch. Keep PRs focused.

### Before Writing Code
- Non-trivial tasks (new features, multi-file changes): use plan mode first.
- New backend resources: use `/go-service-scaffold`.
- New endpoints on existing resources: use `/api-endpoint`.
- Database schema changes: use `/db-migration` first.
- New frontend pages: use `/frontend-feature`.

### Quality Gates (MUST Pass Before Push)
Backend:
```bash
cd backend && go vet ./... && go test -race ./...
```
Frontend:
```bash
cd frontend && npm run lint && npm run test && npm run build
```
E2E:
```bash
cd frontend && npm run e2e
```
Run only gates for packages you changed. Do NOT push code that fails.

Hooks run automatically — `pre-commit` (lint-staged), `commit-msg` (commitlint), `pre-push` (quality gates per changed package).

### Code Review (MUST Run Before Creating PR)
- Backend changes: run `/go-review`.
- Changes touching auth, middleware, user input, or API keys: run `/security-review`.

### Pull Requests
- Create via `gh pr create`. Use conventional commit title (e.g., `feat: add webhook support`).
- Squash merge only. Include Summary and Test Plan in PR body.
- Wait for `ci-gate` to pass before merging.

### After Merge
- Check `deploy-verify` job in GitHub Actions to confirm deploy succeeded.
- `post-deploy-e2e` runs smoke tests (`@smoke` tag) against production after deploy.
- If failed, check Railway logs immediately.

### Commit Messages
Format: `type: short description` (lowercase, no period, max 72 chars)
Types: `feat`, `fix`, `chore`, `refactor`, `ci`, `docs`, `test`

## Architecture

### Backend — Clean Architecture (Go)

Dependency flow: `handler → service → repository → database`

- **`cmd/server/main.go`** — Entry point. Wires up config, pgxpool, router, graceful shutdown.
- **`internal/config/`** — Env-based config using `caarlos0/env`. Loads `.env` via godotenv.
- **`internal/domain/`** — Pure domain structs (Customer, Pixel, PixelEvent, ReplaySession). No dependencies.
- **`internal/repository/interfaces.go`** — Repository interfaces (CustomerRepository, PixelRepository, EventRepository, etc.).
- **`internal/repository/postgres/`** — pgx/v5 implementations. Hand-written SQL (not using sqlc-generated code in repos — sqlc is set up but repos use direct queries).
- **`internal/service/`** — Business logic. Each service takes repository interfaces. Sentinel errors (ErrPixelNotFound, etc.) for handler error mapping.
- **`internal/handler/`** — HTTP handlers. Use `go-playground/validator` for input validation. Consistent response via `handler.JSON()` / `handler.ErrorJSON()` with `APIResponse` / `PaginatedResponse` structs.
- **`internal/middleware/`** — JWT auth (Bearer token, HMAC-SHA256), API key auth (X-API-Key header), CORS, request ID, structured logging.
- **`internal/router/router.go`** — chi router. All DI wiring happens here (repos → services → handlers).
- **`internal/facebook/capi.go`** — Facebook Conversions API HTTP client.

### Authentication — Dual Mode
- **Dashboard (JWT)**: `POST /api/v1/auth/register|login|refresh` → Bearer token. Customer ID stored in JWT `sub` claim, extracted via `middleware.GetCustomerID(ctx)`.
- **Sale Pages (API Key)**: `X-API-Key` header → validated against `customers.api_key` column. Used by sale page templates for event ingestion.

### Key Patterns
- Ownership verification: Services always check `pixel.CustomerID == customerID` before operations.
- CAPI forwarding: Events are forwarded to Facebook asynchronously via `go s.forwardToCAPI(...)`.
- Replay: Creates a background goroutine with semaphore-limited concurrency (5 workers), ~50 events/sec rate limit, progress tracking.
- Repository nil convention: `GetByID` returns `(nil, nil)` when not found (not an error), callers check for nil.

### Frontend Architecture
- **State**: Zustand for auth state, TanStack Query for server state
- **Routing**: react-router v7 with `ProtectedRoute` wrapper
- **API client**: Axios with auto token refresh interceptor (`src/lib/api.ts`)
- **Path alias**: `@/` maps to `frontend/src/`
- **UI components**: shadcn/ui pattern in `src/components/ui/`
- **Vite proxy**: `/api` requests proxy to `http://localhost:8080` in dev

## API Routes

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/v1/auth/register` | Public | Register |
| POST | `/api/v1/auth/login` | Public | Login |
| POST | `/api/v1/auth/refresh` | Public | Token refresh |
| POST | `/api/v1/events/ingest` | API Key | Sale page event ingestion (batch) |
| GET/POST/PUT/DELETE | `/api/v1/pixels/*` | JWT | Pixel CRUD |
| GET | `/api/v1/events` | JWT | Event log (paginated) |
| POST/GET | `/api/v1/replays/*` | JWT | Replay sessions |
| GET | `/api/v1/analytics/*` | JWT | Dashboard analytics |

## Database

PostgreSQL on Neon. 5 active tables: `customers`, `pixels`, `pixel_events`, `replay_sessions`, `refresh_tokens`. (`event_rules` table exists but is unused.) All IDs are UUID with `gen_random_uuid()`. Timestamps are `TIMESTAMPTZ`.

## Deployment

Railway with separate services:
- **Backend**: `backend/Dockerfile` — Multi-stage Go build, Alpine-based
- **Frontend**: `frontend/Dockerfile` — Node build → Nginx serving static files. Requires `VITE_API_URL` build arg.

## Environment Variables

Backend (see `backend/.env.example`):
- `DATABASE_URL` (required) — Neon PostgreSQL connection string
- `JWT_SECRET` (required) — HMAC signing key
- `PORT`, `ENV`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`, `FB_GRAPH_API_URL`, `CORS_ALLOWED_ORIGINS`, `RATE_LIMIT_RPS`

Frontend:
- `VITE_API_URL` — Backend API base URL (empty string uses relative path with Vite proxy in dev)
